PR description:
## Problem

The PostHog Slack app shares the same chat agent system prompt as the web app, and that prompt has no length guidance — so answers in Slack came out as long as web answers. Slack readers want short, skimmable replies, and users reported that even a few bullet points read as too long for a chat thread.

**Why:** direct user feedback that the Slack bot is too verbose — a one-line answer to a simple question should stay one line.

## Changes

Adds a Slack-only response-style block to the chat agent system prompt:

- Defaults to a **single sentence** for simple questions; brief by default otherwise.
- Treats bullet lists as the exception, not the norm; prefers plain prose over headers and tables.
- Drops preamble, question restatement, and sign-offs.
- Framed as a default, not a hard rule — if the user (or their saved memory) asks for more depth or a different format, that still wins.

The block is injected **only when the request originates from a Slack thread** (detected via `slack_thread_context` in the run config), so the web app is unaffected. A new `_get_extra_system_prompts(config)` hook on the base prompt builder carries it, and both the chat and plan builders opt in.

## How did you test this code?

I'm an agent. I added an automated parameterized test (`test_prompt_builder_includes_slack_response_style_only_from_slack`) asserting the block is present when a Slack thread context is in the config and absent otherwise, for both the chat and plan prompt builders. I did not run the suite — the Django/ClickHouse test environment wasn't available in this session — but all changed files compile.

def _slack_response_style_prompts(config: RunnableConfig) -> list[str]:
    """Slack threads want shorter, skimmable answers than the web app, so nudge the agent to be
    concise only when the request actually originates from Slack."""
    if (config.get("configurable") or {}).get("slack_thread_context") is not None:
        return [SLACK_RESPONSE_STYLE_PROMPT]
    return []
